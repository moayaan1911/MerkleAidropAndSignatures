üë®‚Äçüè´ Normal Explanation

To save gas (money), we don't store the list of 10,000 eligible users on the blockchain. Instead, we use a Merkle Tree.

1. We "zip" the entire list into one unique code called the Merkle Root and store only that on-chain.
2. When a user wants to claim, they provide a Proof (the path through the tree) showing they are part of that Root.
3. We also need to be sure the person claiming is actually the owner of the address, so we check their digital Signature.

üë∂ ELI5: The "Golden Ticket" Analogy

Imagine an Exclusive Party with 10,000 invited guests.

Instead of giving the bouncer a huge book with all 10,000 names (which would be super heavy and take forever to check), the club owner gives the bouncer just ONE Magic Stamp.

To get in, you need two things:

  • üé´ The Golden Ticket: A special piece of paper that, when you fold it in a certain way and show it to the bouncer, magically matches the Magic Stamp. This proves you were invited without the bouncer needing to check a big list!
  • ‚úçÔ∏è Your Secret Handshake: Even if you have the ticket, you need to do a special handshake that only YOU know. This proves it's really you and not someone who stole your ticket!

Why is this cool? The bouncer doesn't need a giant book, just one tiny stamp. And you can prove you're invited without showing the entire guest list to everyone!

2. Files Deep Dive üìÇ

üìú MerkelAirdrop.sol

The brain of the operation - handles all claim logic and security checks

Core Contract

üìç Purpose

This is the main smart contract that lives on the blockchain. It stores the Merkle Root (one compressed fingerprint of all eligible users) and handles the token distribution. When someone wants to claim their tokens, this contract performs THREE critical security checks before sending anything.

üîë Key Components

üóÑÔ∏è State Variables
  • i_merkleRoot - The ONE magic number that represents all 10,000 eligible users
  • i_airdropToken - Reference to the BagelToken contract (what we're giving away)
  • s_hasClaimed - A checklist mapping: has this address claimed yet? (prevents double-dipping)
üèóÔ∏è Constructor

When we deploy this contract, we pass in the Merkle Root (calculated off-chain) and the token address. These are stored as immutable, meaning they can NEVER be changed. The list is sealed forever!

‚ö° The claim() Function - Step by Step

src/MerkelAirdrop.sol - claim()
function claim( address _account, uint256 _amount, bytes32[] calldata _merkleProof, uint8 _v, bytes32 _r, bytes32 _s ) external { // ‚úÖ CHECK 1: Has this person already claimed? if (s_hasClaimed[_account]) revert AlreadyClaimed(); // ‚úÖ CHECK 2: Is the signature valid? (Proves ownership) if (!_isValidSignature(_account, getMessageHash(_account, _amount), _v, _r, _s)) { revert InvalidSignature(); } // ‚úÖ CHECK 3: Is this person on the whitelist? (Merkle proof) bytes32 leaf = keccak256(bytes.concat(keccak256(abi.encode(_account, _amount)))); if (!MerkleProof.verify(_merkleProof, i_merkleRoot, leaf)) { revert InvalidProof(); } // ‚úÖ MARK as claimed s_hasClaimed[_account] = true; // üí∞ SEND THE TOKENS! emit AirdropClaimed(_account, _amount); i_airdropToken.safeTransfer(_account, _amount); }
1
Double-Claim Prevention üö´

We check the s_hasClaimed mapping. Think of it as a giant checklist. If your address already has a checkmark, the transaction fails immediately. This prevents someone from claiming multiple times.

2
Signature Verification ‚úçÔ∏è

This is where we prove YOU are actually YOU. The contract takes the message (your address + amount) and creates a special code using EIP-712 standard. Then it uses the signature components (v, r, s) to mathematically recover the address that signed this message.

The Magic: The ecrecover function (inside _isValidSignature) spits out an address. If that address matches _account, we know the signature is legit! This works without ever seeing your private key.

3
Merkle Proof Verification üå≥

This is where we check if you're on the whitelist WITHOUT storing the entire list on-chain! Here's how:

  1. We take your data (address + amount) and convert it into a leaf (a 32-byte fingerprint)
  2. You provide a proof - an array of other fingerprints (your "path" through the tree)
  3. The contract combines your leaf with the proof pieces, step by step, climbing up the tree
  4. If we reach the i_merkleRoot stored in the contract, you're verified!

Why double hash? Notice we hash TWICE: keccak256(bytes.concat(keccak256(...))). This prevents a sneaky attack where someone could fake a leaf by combining two other nodes. The first hash standardizes the data, the second hash protects against manipulation.

4
Token Transfer üí∞

All checks passed! We mark you as claimed (so you can't claim again), emit an event for transparency, and finally call safeTransfer to send the tokens to your address. The safeTransfer function from OpenZeppelin ensures the transfer succeeds or the entire transaction reverts.

3. Digital Signatures Explained ‚úçÔ∏è

‚úçÔ∏è

What are v, r, s?

The Three Magic Numbers of Every Ethereum Signature

üéØ The Big Picture

When you "sign" something in Ethereum, you're not writing your name. You're doing complex math on something called an Elliptic Curve using your secret Private Key. The output is three numbers that together prove "I own this address and I approved this message."

r
The X-Coordinate

This is the X position of a random point on the elliptic curve. When you sign, you generate a random number, multiply it on the curve, and take the X coordinate. This is your r value.

s
The Proof

This is calculated using your private key, the message you're signing, and the random number from r. It's the mathematical "proof" that ties everything together. Without your private key, you can't create a valid s.

v
The Selector (Recovery ID)

The elliptic curve is symmetric - like a butterfly. For any X coordinate, there are TWO possible points (one above the middle line, one below). The v value (either 27 or 28) tells us which of the two points to use. This helps recover your public key from the signature.

Elliptic Curve Visualization

‚Ä¢ The blue curve is the secp256k1 elliptic curve used by Ethereum

‚Ä¢ r is the X-coordinate (the vertical red line)

‚Ä¢ There are TWO points with the same X: one above, one below

‚Ä¢ v (27 or 28) tells us which point to use

‚Ä¢ s is calculated using your private key (not shown on graph)

üë®‚Äçüè´ How Signature Verification Works

Step 1: Signing (Off-Chain)

You have a message ("I claim 25 tokens"). You take your Private Key and the message, run them through ECDSA math, and get three numbers: v, r, s. This happens in your wallet (MetaMask, Ledger, etc.)

Step 2: Submission (On-Chain)

You (or a relayer) send the message + signature (v, r, s) to the smart contract. The contract doesn't know your private key and never will.

Step 3: Recovery (On-Chain)

The contract uses ecrecover(messageHash, v, r, s). This function does reverse math and outputs an Ethereum address.

Step 4: Verification (On-Chain)

If the recovered address matches the claiming address, the signature is valid! This proves the claimer owns the private key without ever revealing it.

üë∂ ELI5: The Secret Handshake

Imagine you and your best friend have a Secret Handshake that only you two know. Nobody else in the world can do it exactly right.

ü§ù The Handshake (v, r, s)

When you want to prove it's really you, you do the handshake. The handshake has three parts: where you start (r), how you move your hands (s), and whether you go high or low (v).

üëÄ The Verification

The teacher watches you do the handshake. Even though the teacher doesn't know the secret, they can tell if you did it correctly by checking if it matches the pattern. If it matches, they know you're the real deal!

üîê The Secret Stays Secret

The amazing part? The teacher never learns your secret handshake! They just know you did it right. Same with blockchain - the contract verifies your signature without ever seeing your secret key.

4. Merkle Trees Explained üå≥

üå≥

What is a Merkle Tree?

The Ultimate Compression Trick for Blockchain

A Merkle Tree is a way to take a HUGE list of data and compress it into ONE tiny fingerprint (32 bytes), while still being able to prove any item was in the original list. It's like having a library of 10,000 books, but instead of storing all the books, you just store ONE magic number that represents them all.

üìä Visual Representation

üéØ How User 2 Proves They're on the List

User 2 doesn't need to show the entire tree! They only need:
1. Their own data (Leaf B: address + 25 tokens)
2. Proof[0]: Leaf A's fingerprint (their sibling)
3. Proof[1]: Hash CD (the other branch)

The contract combines these step by step: Hash(Leaf A + Leaf B) = Hash AB, then Hash(Hash AB + Hash CD) = ROOT. If it matches the stored root, User 2 is verified! ‚úÖ

üë®‚Äçüè´ Technical Explanation

Building the Tree

Start with all users at the bottom (leaves). Pair them up and combine each pair into a parent. Keep pairing and combining until you reach one final piece at the top (root). Each "combination" is done by putting two pieces together and scrambling them into a new unique code.

Storing the Root

Instead of storing 10,000 addresses on-chain (expensive!), we only store the ROOT (32 bytes). This one number represents the entire tree.

Proving Membership

To prove you're in the tree, you provide your leaf data + the "path" (sibling pieces needed to climb to the root). The contract rebuilds the path and checks if it matches the stored root.

Why It's Secure

You can't fake a proof because changing even ONE character in your data completely changes the final root. The math makes it impossible to forge.

üë∂ ELI5: The LEGO Tower

Imagine you have 10,000 different colored LEGO bricks on the ground. You want to prove one specific red brick is in your collection.

üß± Building the Tower

You take two bricks and glue them together with special glue that turns them into ONE new brick with a new color. Then you take two of those new bricks and glue them together. Keep going until you have ONE brick at the very top. That top brick is the ROOT!

üì∏ Taking a Picture

Instead of carrying around all 10,000 bricks, you just take a picture of the TOP brick (the root). That one picture represents your entire collection!

üîç Proving You Have the Red Brick

To prove you have the red brick, you show: (1) the red brick, (2) the brick it was glued to, (3) the brick that pair was glued to, and so on. Someone can follow your path up the tower and see if they reach the same top brick in the picture. If yes, you proved it!

‚ú® The Magic

You only showed a few bricks (the path), not all 10,000! And you only stored one picture (the root), not all 10,000 bricks! That's the magic of Merkle Trees - tiny storage, easy proof!

MakeMerkle.s.sol

üå≥ The Merkle Tree Builder

forge script

üéØ Purpose

This is the MOST IMPORTANT off-chain step. It reads the whitelist from input.json, performs heavy computation to build the Merkle Tree, and generates the Root + individual Proofs for each user. This is where the magic compression happens!

‚Üí What It Does (Step by Step)
Step 2.1: Read Input

Uses vm.readFile() to load the entire input.json as a string, then parses it using Foundry's stdJson library.

Step 2.2: Convert to Leaves

For each whitelist entry, it converts the address and amount into a bytes32 leaf using the SAME double-hash logic as the contract: keccak256(bytes.concat(keccak256(abi.encode(...)))). This ensures consistency!

Step 2.3: Build the Tree

Uses the Murky library (a Foundry Merkle tree library) to construct the tree from all leaves. Murky handles the pairing and hashing automatically.

Step 2.4: Generate Proofs

For EACH user, it calls m.getProof(leafs, i) to get the array of sibling hashes needed to prove that leaf is in the tree. This is the "path" we talked about!

Step 2.5: Calculate Root

Calls m.getRoot(leafs) to get the final Merkle Root. This is the ONE number we'll store on-chain!

Step 2.6: Write Output

Creates a JSON file with all the data: for each user, it stores their inputs, proof array, root, and leaf hash. This file is what users (or relayers) will use to claim tokens.

‚Üí How to Run It
forge script script/MakeMerkel.s.sol

Again, no blockchain interaction - just heavy computation and file writing.

‚Üí What It Does (Detailed Breakdown)

Transfers all 100 tokens from the deployer to the MerkelAirdrop contract using bagelToken.transfer(address(merkelAirdrop), 100e18). Now the airdrop contract has tokens to give away!

Interact.s.sol (ClaimAirdrop)

üí∞ The Claim Script

forge script --broadcast

üéØ Purpose

This script simulates a user claiming their airdrop. In production, users would do this through a frontend (dApp), but for testing, we use this script. It acts as a "relayer" - it has the user's signature and submits the claim transaction on their behalf.

‚Üí What It Does (The Final Steps!)
Step 4.1: Find Deployed Contract

Uses DevOpsTools.get_most_recent_deployment() to automatically find the latest MerkelAirdrop contract address on the current chain. This is super convenient - no need to manually copy-paste addresses!

Step 4.2: Load Proof from output.json

The script has the Merkle proof hardcoded (extracted from output.json for User 2). In a real dApp, the frontend would fetch this from a database or IPFS based on the connected wallet address.

The script contains a pre-generated signature (65 bytes: r + s + v). This was created OFF-CHAIN by signing the message with the user's private key. The splitSignature() function breaks it into v, r, s components.

Finally! We call merkelAirdrop.claim(address, amount, proof, v, r, s). The contract:

Step 4.5: Success!

If all checks pass, the user receives their tokens! The contract emits an AirdropClaimed event, and the user's balance increases by 25 BAGEL. They can now use, transfer, or sell their tokens!
